# agent

ユーザーが「agent を追加したい」と種別を確信しているときに、種別判定を省略して agent author agent を直叩きする最短経路。`/forge:new` の planner 経由フローを通すと判定プロセスが冗長になるケースを救う。

## When to Activate

- ユーザーが `/forge:agent` を実行したとき
- 「PR レビュー担当の agent が欲しい」「セキュリティレビュアーを作りたい」「実装プランナーが欲しい」と種別を明示したとき
- ペルソナ + 重要度判定（CRITICAL/HIGH/MEDIUM）または Phase ワークフローで構造化される要望

種別が不明・組合せかもしれない要望は `/forge:new`（planner 経由）に誘導する。

## 設計思想

- 種別が確定しているなら planner を通さない。判定の往復は精度に貢献せず、時間とコンテキストを消費するだけになる。
- バリエーション判定（レビュー / プランナー / リゾルバー）は agent author の専門領域なので、本 skill では深入りしない（主題から強い手がかりがあれば伝える程度）。
- ユーザーが種別を誤認している可能性は本 skill 側で粗く検出する。「詳細手順を中心に書きたい」「絶対やらないリスト中心」のように agent の責務外が要望の中心なら、別コマンドへ誘導する。

## アーキテクチャ

```
ユーザー要望（種別 = agent 確定）
    │
    ▼
[agent skill]（本 skill）
    │
    ▼
[agent author agent] ──▶ .claude/agents/{name}.md
                              │
                              ▼
              path-scoped [rules/agents.md] が自動適用
```

planner agent は呼ばない。

## 配置先

| 対象 | デフォルトパス |
|------|---------------|
| agent | `.claude/agents/<kebab-name>.md` |

プロジェクト固有の配置規約があれば author 委譲前にユーザーへ確認する。

## 設定

| 環境変数 | デフォルト | 説明 |
|---------|----------|------|
| FORGE_AGENT_VARIANT_HINT | auto | `auto` / `review` / `planner` / `resolver` -- バリエーション初期ヒント |

## ワークフロー

### Step 1: 主題とバリエーションの粗い判定

ユーザー発話から「役割」「主たる動詞（レビュー / 計画 / 修正）」「対象領域」を抽出する。動詞からバリエーションの初期ヒントを決める。

| 主動詞 | 初期ヒント |
|-------|-----------|
| レビュー / 検査 / 判定 | A. レビュー系（CRITICAL/HIGH/MEDIUM/LOW） |
| 計画 / 設計 / 提案 | B. プランナー系（Phase 1/2/3） |
| 修正 / 解消 / 復旧 | C. リゾルバー系（DO/DON'T + 最小 diff） |

判定はあくまで初期値。最終決定は author の責務。

### Step 2: 越境チェック（軽量）

要望文に agent author の責務外が強く含まれていないか確認する。

| 兆候 | 対処 |
|------|------|
| 「詳細な手順を一通り書いてほしい」「PASS/FAIL コード例が中心」 | `/forge:skill` を提案 |
| 「常にやる / 絶対やらない」が主目的（短い宣言で済む） | `/forge:rule` を提案 |
| 「ペルソナ + 詳細手順 + ガードレール一式」が混ざる | `/forge:new`（planner 経由）を提案 |

該当しなければそのまま Step 3 へ進む。該当した場合はユーザーに切替を提案し、承認を得てから委譲する。

### Step 3: agent author に委譲

`agent` agent に以下を渡す:

- 主題（ユーザー発話の要点）
- 配置先パス
- バリエーション初期ヒント（`FORGE_AGENT_VARIANT_HINT` および Step 1 の判定）
- 既存衝突確認の指示（配置先パスを Read してエラー有無で判定）

### Step 4: 結果レポート

author の出力をそのままユーザーに提示する。最低限の集約のみ行う。

```
## 生成結果
- パス: {path}
- バリエーション: {A レビュー / B プランナー / C リゾルバー}
- model: {opus / sonnet / haiku}
- tools: {...}
- 次のアクション: {関連 skill / rule のクロスリンク追加など}
```

## エラーハンドリング

- **既存ファイルと衝突**: 上書きせず、`/forge:refactor` を提案する
- **author が選んだ model / tools が要望と乖離**: ユーザーに確認する。再委譲は最大 1 回
- **要望が agent の責務外を多く含む**: Step 2 で別コマンドへ誘導する。無理に agent として書かせない

## アンチパターン

- **越境チェックを省略して即委譲**: 種別誤認をそのまま固定化する。Step 2 は短くても必ず通す
- **planner を念のため呼ぶ**: 本 skill の存在意義を消す。planner が必要なら `/forge:new` を使う
- **バリエーションを本 skill で確定させる**: 初期ヒントに留め、最終判定は author に委ねる
- **tools を勝手に決めて author に強制する**: 最小権限の原則は author 側のテンプレ知識で担保される

---

**Remember**: 種別が決まっているなら最短経路で書く。planner は迷ったときの調停役、確信があるときは通さない。
