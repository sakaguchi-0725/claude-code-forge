# rule

ユーザーが「rule を追加したい」と種別を確信しているときに、種別判定を省略して rule author agent を直叩きする最短経路。`/forge:new` の planner 経由フローを通すと判定プロセスが冗長になるケースを救う。

## When to Activate

- ユーザーが `/forge:rule` を実行したとき
- 「常に〜する」「絶対に〜しない」を宣言として残したい要望
- 言語・ディレクトリ固有のガードレール（`paths:` フロントマターでスコープ）を追加したい要望

種別が不明・組合せかもしれない要望は `/forge:new`（planner 経由）に誘導する。

## 設計思想

- 種別が確定しているなら planner を通さない。判定の往復は精度に貢献せず、時間とコンテキストを消費するだけになる。
- rule は短い（20〜80 行）ので越境すると影響が大きい。手順や長いコード例が混ざると一気に skill に化けるので、本 skill 側の越境チェックは特に厳しめにする。
- `paths:` スコープの判定は author の責務。本 skill では「全体共通か / 言語固有か / ディレクトリ固有か」の手がかりだけ集める。

## アーキテクチャ

```
ユーザー要望（種別 = rule 確定）
    │
    ▼
[rule skill]（本 skill）
    │
    ▼
[rule author agent] ──▶ .claude/rules/{name}.md
                              │
                              ▼
              path-scoped [rules/rules.md] が自動適用
```

planner agent は呼ばない。

## 配置先

| 対象 | デフォルトパス |
|------|---------------|
| rule | `.claude/rules/<kebab-name>.md` |

プロジェクト固有の配置規約があれば author 委譲前にユーザーへ確認する。

## 設定

| 環境変数 | デフォルト | 説明 |
|---------|----------|------|
| FORGE_RULE_SCOPE_HINT | auto | `auto` / `global` / `lang` / `dir` -- スコープ初期ヒント |

## ワークフロー

### Step 1: 制約とスコープの抽出

ユーザー発話から以下を抽出する。

- 制約本体（ALWAYS / NEVER / MUST のいずれに当たるか）
- スコープ範囲（全体共通 / 特定言語 / 特定ディレクトリ）
- 関連する agent / skill（クロスリファレンス候補）

スコープが言語・ディレクトリ固有だが要望に明示されていない場合は 1 問だけ確認する。

### Step 2: 越境チェック（厳しめ）

rule は短いので、要望に他種別の要素が混ざっていると特に影響が大きい。以下を厳しく判定する。

| 兆候 | 対処 |
|------|------|
| 「手順を 1, 2, 3, ... と書いてほしい」 | `/forge:skill` を提案（手順は rule の責務外） |
| 「PASS / FAIL コード例で説明したい」 | `/forge:skill` を提案（OK/NG 1 組までは rule で許容するが、それ以上は skill） |
| 「ペルソナとして〜役を立てたい」 | `/forge:agent` を提案 |
| 「ガードレール + 手順 + レビュー観点」が混ざる | `/forge:new`（planner 経由）を提案 |
| 具体的な正規表現・トークン接頭辞・ファイルグロブを列挙したい | 列挙部分を skill に逃がし、rule には抽象的な制約のみ残す案を提示 |

該当しなければ Step 3 へ進む。

### Step 3: rule author に委譲

`rule` agent に以下を渡す:

- 制約本体（ALWAYS / NEVER / MUST に分類した bullet）
- スコープ（`paths:` の初期値、全体共通なら省略）
- 関連 agent / skill のリスト（クロスリファレンス先）
- 既存衝突確認の指示（配置先パスを Read してエラー有無で判定）

### Step 4: 結果レポート

author の出力をそのままユーザーに提示する。最低限の集約のみ行う。

```
## 生成結果
- パス: {path}
- paths スコープ: {値 or 共通}
- ALWAYS 件数: {n}
- NEVER 件数: {n}
- 関連参照: {agent / skill 数}
- 次のアクション: {関連 agent / skill 側のクロスリンク追加など}
```

## エラーハンドリング

- **既存ファイルと衝突**: 上書きせず、`/forge:refactor` を提案する
- **author が rule を 80 行超で生成**: テンプレ違反。再委譲して手順部分を skill に逃がすよう指示（最大 1 回）
- **要望が rule の責務外を多く含む**: Step 2 で別コマンドへ誘導する。無理に rule として書かせない

## アンチパターン

- **越境チェックを省略して即委譲**: rule は短いので、越境した瞬間にテンプレ違反が顕在化する。Step 2 は必ず通す
- **planner を念のため呼ぶ**: 本 skill の存在意義を消す。planner が必要なら `/forge:new` を使う
- **`paths:` を勝手に確定させる**: スコープは制約の本質に関わる。曖昧ならユーザーに確認する
- **関連 agent / skill のクロスリファレンスを省略**: 詳細を委譲できないと rule に手順が膨らむ温床になる

---

**Remember**: 種別が決まっているなら最短経路で書く。planner は迷ったときの調停役、確信があるときは通さない。
