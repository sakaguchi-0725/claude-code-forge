# refactor

既存の設定ファイルを読み込み、種別が正しいかを検証してから、テンプレート準拠に書き換えるハーネス系 skill。

## When to Activate

- ユーザーが `/forge:refactor` を実行したとき
- 「この設定ファイルをテンプレート準拠にしたい」と対象を指定されたとき
- 「この skill / agent / rule を整えて」と既存ファイルを渡されたとき
- 既存設定の「種別が正しいか不安」という相談を受けたとき

## 設計思想

リファクタでは「書き換え」の前に「種別検証」が必要。例えば skill として置かれているが中身が agent（ペルソナ定義から始まる）というケースがある。

- 種別を固定して書き換えると、中身に合わない型に無理やり押し込むことになる
- まず planner に「この内容は本当にこの種別か？」を問わせる
- 種別変更が必要な場合はファイル移動（削除 + 新規作成）を伴うため、必ずユーザー確認を挟む
- 種別が正しい場合は対応する author agent が path-scoped rule の下で書き換える

## アーキテクチャ

```
対象ファイル
    │
    ▼
[refactor skill]（本 skill）
    │
    ▼
[Read で全文取得] + [Glob で周辺資産確認]
    │
    ▼
[planner agent] -- 種別検証
    │
    ├── 種別正しい ──▶ [{kind} agent] ──▶ 同一パスに書き換え (Edit)
    └── 種別ずれ ────▶ ユーザー確認
                        │
                        ├─▶ 承認 ──▶ [{new-kind} agent] + 旧ファイル削除
                        └─▶ 却下 ──▶ 中断
```

## 設定

| 環境変数 | デフォルト | 説明 |
|---------|----------|------|
| FORGE_CONFIRM_KIND_CHANGE | true | 種別変更時にユーザー確認を必須とするか |
| FORGE_KEEP_OLD_ON_MOVE | false | 種別変更時に旧ファイルを残すか（デフォルトは削除前に確認） |

## ワークフロー

### Step 1: 対象ファイルの特定と読み込み

配置パスから現状の種別を推定する。

```bash
# .claude/skills/*/SKILL.md → skill
# .claude/agents/*.md       → agent
# .claude/rules/*.md        → rule
```

Read で全文を取得する。また Glob で同ディレクトリの周辺ファイルを確認し、命名・構造の整合性を後段で判定できるようにする。

### Step 2: planner agent で種別検証

planner に以下を渡す:

- 現在のファイルパス（= 現状の種別の推定値）
- ファイル内容全文
- 周辺ファイルのリスト

planner の返答:

- **種別一致**: そのまま Step 4 の書き換えへ
- **種別不一致**: 移動先パス案と理由をユーザーに提示（Step 3 へ）
- **いずれの種別にも当てはまらない**: 「この内容は skill / agent / rule のどれでもない」と判定されたら、削除 or 他形式への変換をユーザーに相談

### Step 3: 種別変更の確認（種別不一致時のみ）

`FORGE_CONFIRM_KIND_CHANGE=true` なら以下をユーザーに確認する。

- 新しい配置先パス
- 旧ファイルの削除可否（`FORGE_KEEP_OLD_ON_MOVE` の値による）
- 内容の欠損リスク（例: skill → rule なら詳細手順が切り落とされる）

承認されなければ中断する。既存ファイルには触れない。

### Step 4: author agent に委譲して書き換え

判定された種別に対応する author agent を呼び出す。入力:

- 現状のファイル内容
- 配置先パス（種別変更時は新パス）
- テンプレート準拠で何を補強すべきか（欠損セクション・違反箇所の一覧）

author は path-scoped な `rules/*.md` が自動適用された状態で書き換えるので、テンプレート違反を避けられる。

### Step 5: 差分レポート

```
## リファクタ結果
- パス: {元 path} {→ 新 path（種別変更時のみ）}
- 種別: {元} {→ 新（種別変更時のみ）}
- 補強したセクション: {...}
- 違反是正: {...}
- 削除した旧ファイル: {path or なし}
```

## エラーハンドリング

- **種別変更で旧ファイル削除が承認されない**: 新ファイル生成のみ行い、旧は残す。ユーザーに後で整理するよう伝える
- **author 書き換え失敗**: 元の内容を変更せず中断する（Write ではなく一旦内容をメモリに保持して検証してから書き込む）
- **テンプレートに当てはまらない内容**: planner が「どの種別でもない」と判定したら、ユーザーに削除 or 他形式（CLAUDE.md 等）への変換を相談する
- **複数ファイルを同時指定**: 1 ファイルずつ順次処理する。中間で失敗しても既に完了した分はロールバックしない（MVP）

## アンチパターン

- **種別検証をスキップして書き換え**: 中身と型のミスマッチを固定化する。必ず planner を経由する
- **種別変更を無断実行**: ファイル削除はユーザーの合意なしで行わない
- **1 ファイルに複数種別を混在**: rule の中に手順を書き足すような「継ぎ足し」はしない。必要なら分割して別ファイルに分ける
- **周辺ファイルの確認を省略**: 命名規則やクロスリファレンスの整合性を落とす原因になる

---

**Remember**: 書き換える前に「これは本当にこの種別か？」を問え。型に合わない内容を無理に収めない。
